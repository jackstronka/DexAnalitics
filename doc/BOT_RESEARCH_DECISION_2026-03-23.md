# Bot Research Decision (2026-03-23)

## Context

Goal of this research phase:
- decide whether to build on top of external bots/frameworks or keep our own runtime as the primary base,
- prioritize an Orca-first path for live deployment,
- ensure strategy semantics match our backtest model, including the new `RetouchShift`.

User priorities:
- research first, implementation second,
- document what was analyzed and why,
- include architecture and operational risks, not only feature lists.

## Requirements Baseline (What We Need)

Must-have:
- 4 strategy semantics: `StaticRange`, `Periodic`, `Threshold`, `RetouchShift`,
- `RetouchShift` behavior: move only exiting edge, keep width, `once_until_back_in`,
- collect fees before range change,
- Mainnet readiness with robust tx handling (retry, priority fees, simulation),
- Orca first; Raydium/Meteora follow-up by adapter architecture.

Nice-to-have:
- reusable controller/executor split,
- policy gates (cooldowns, rate limits, drawdown guards),
- clean lifecycle/event trail for operations and diagnostics.

## Comparative Matrix

### 1) Hummingbot V2 (`controllers + executors`)
- Fit for CLMM auto-rebalance: High
- Fit for our exact `RetouchShift`: Medium (requires custom policy/controller logic)
- Operational maturity: High
- Integration cost into current Rust stack: High
- Risk:
  - framework coupling,
  - adaptation overhead for Solana-specific production flow.
- Key value to adopt:
  - controller -> executor action architecture,
  - config-driven strategy orchestration,
  - policy-oriented runtime split.

### 2) Orca Whirlpools Repositioning Bot (PR #558)
- Fit for Orca-first live deployment: Very High
- Fit for our strategy model: Medium-High
- Solana operational quality (priority fee, retry, transaction flow): High
- Multi-protocol portability: Low
- Key value to adopt:
  - Orca-centric tx lifecycle practices,
  - practical failure handling,
  - close/open reposition flow patterns.

### 3) RangeKeeper (7-band model)
- Continuous-liquidity design strength: High
- Fit for single-position `RetouchShift`: Low-Medium
- Complexity/cost: High
- Key value to adopt:
  - concept of avoiding liquidity gaps during repositioning.

### 4) Gelato / G-UNI
- Checker/upkeep automation quality: High
- Direct Solana portability: Low-Medium
- Key value to adopt:
  - explicit condition checker before execution,
  - scheduler + threshold gates as first-class architecture.

### 5) Verisana / Vasily (legacy Uniswap examples)
- Educational value: Medium
- Production readiness today: Low
- Key value to adopt:
  - historical patterns only; not suitable as implementation base.

### 6) Arrakis (vault manager pattern)
- Architecture quality: High
- Direct fit to current Orca-first bot scope: Medium
- Key value to adopt:
  - long-term manager/vault abstraction concepts.

## Decision

### Recommended Direction
- Keep our codebase as primary runtime (backtest + execution + API).
- Use external bots/frameworks as architecture references, not as direct runtime dependency.
- Prioritize Orca operational hardening first, then generalize adapter boundaries for Raydium/Meteora.

### Why
- Best time-to-value for current product scope.
- Lowest integration risk with existing domain/backtest logic.
- Highest control over exact strategy semantics (`RetouchShift`, gating, fee-before-rebalance).

## What To Adopt From External Research

Adopt now:
- Controller/decision layer separated from executor/tx layer.
- Explicit checker/policy stage (cooldown, trigger gating, drawdown/risk gates).
- Solana tx best practices: simulation-aware send path, retry/backoff, priority-fee policy.

Do not adopt now:
- full framework migration to external bot runtime,
- complex multi-band vault architecture before single-market Orca path is stable.

## Risks and Mitigations

Risk: strategy drift between backtest and live.
- Mitigation: keep one strategy semantics source and map live decisions 1:1 to backtest behavior.

Risk: Solana tx reliability issues in live loop.
- Mitigation: retries, simulation checks, explicit failure taxonomy, and operational runbook.

Risk: scope creep from multi-protocol support too early.
- Mitigation: Orca-only live runtime in first stage; adapters for others as follow-up.
